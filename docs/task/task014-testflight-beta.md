# Task 014: TestFlight ベータテスト

## 基本情報

| 項目 | 内容 |
|------|------|
| Phase | 8 - TestFlight ベータテスト |
| 工数 | 2時間 + テスト期間1週間 |

## 目的

TestFlight を使用してベータ版を配信し、内部テスターからフィードバックを収集する。重大なバグを発見・修正してからApp Store本申請に進む。

---

## 📚 学習ポイント：TestFlight とは

### なぜベータテストが重要か？

```
【リリース前テストの重要性】

開発中のテスト → 開発者の視点でのみテスト
                  ↓ 限界がある
              「自分では気づかない問題」が残る

ベータテスト → 実際のユーザーがテスト
              ↓ 発見できる
              ・操作方法の分かりにくさ
              ・開発環境では再現しない不具合
              ・想定外の使い方で起きるバグ

【TestFlight を使う理由】

1. Apple 公式のベータ配信ツール
   - App Store と同じ配信インフラ
   - 安全で信頼性が高い

2. 簡単な参加方法
   - テスターは TestFlight アプリをインストールするだけ
   - 招待リンクをタップ → 即参加可能

3. フィードバック収集機能
   - スクリーンショット付きの報告
   - クラッシュログの自動収集
```

### TestFlight の仕組み

```
【ベータ配信の流れ】

開発者                    Apple                   テスター
  │                        │                        │
  │ 1. ビルド提出           │                        │
  │ ──────────────────────>│                        │
  │                        │                        │
  │                        │ 2. 自動審査             │
  │                        │ （通常24時間以内）      │
  │                        │                        │
  │ 3. 審査完了通知          │                        │
  │ <──────────────────────│                        │
  │                        │                        │
  │ 4. テスター招待          │                        │
  │ ─────────────────────────────────────────────>│
  │                        │                        │
  │                        │ 5. テスト参加           │
  │                        │ <─────────────────────│
  │                        │                        │
  │ 6. フィードバック受信    │                        │
  │ <─────────────────────────────────────────────│
  │                        │                        │
```

### 内部テスターと外部テスターの違い

```
【内部テスター】
- 対象: Apple Developer チームのメンバー
- 人数: 最大100名
- 審査: 不要（即座に配信可能）
- 用途: 開発チーム、社内テスト

【外部テスター】
- 対象: 一般ユーザー
- 人数: 最大10,000名
- 審査: Apple による簡易審査あり（24〜48時間）
- 用途: 大規模ベータテスト、一般公開テスト

【どちらを使うべき？】
MVP段階 → 内部テスター（少人数で密なフィードバック）
本格運用 → 外部テスター（多くのユーザーで検証）
```

### ベータテストで見つけるべき問題

```
【優先的に発見したい問題】

1. クラッシュ・フリーズ
   - アプリが強制終了する
   - 画面が固まって操作できない
   → リリース前に必ず修正

2. データの不整合
   - 学習履歴が消える
   - 進捗が正しく保存されない
   → ユーザー体験を大きく損なう

3. 機能の不具合
   - ボタンが反応しない
   - 画面遷移がおかしい
   → 基本機能は完全に動作すべき

4. UI/UX の問題
   - 文字が小さくて読めない
   - 操作方法が分からない
   → テスターの感想から発見
```

## 📚 学習ポイント：Export Compliance（輸出コンプライアンス）

### 暗号化と輸出規制

```
【なぜ暗号化の申告が必要？】

歴史的背景:
  暗号技術は「武器」として国際規制の対象
  アメリカから暗号技術を「輸出」するには許可が必要

アプリの場合:
  HTTPS通信 → SSL/TLS暗号化を使用
  → 技術的には「暗号化を使用している」

【MVPアプリの場合の回答】

Q1: 暗号化を使用していますか？
A1: はい（HTTPS通信を使っている）

Q2: どのカテゴリ？
A2: 一般に入手可能な暗号化
   （標準的なHTTPS通信のみ）

Q3: 輸出許可が必要？
A3: いいえ（標準的な暗号化は免除対象）

【免除される場合】
✅ HTTPS/TLS 通信のみ
✅ 一般に入手可能な暗号化ライブラリ
✅ 認証のためだけの暗号化

【申告が必要な場合】
❌ 独自の暗号化アルゴリズム
❌ 高度なセキュリティ機能
❌ 政府機関向けの暗号化
```

## 📚 学習ポイント：効果的なフィードバック収集

### フィードバックの種類と対応

```
【フィードバックの分類】

定量的フィードバック:
  - 5段階評価（使いやすさ、見やすさ等）
  - 数値で比較・改善度を測定可能
  - 例: 「使いやすさ: 3.5 → 4.2 に改善」

定性的フィードバック:
  - 自由記述コメント
  - 具体的な問題点・要望を把握
  - 例: 「問題の文字が小さい」

バグレポート:
  - 再現手順、スクリーンショット
  - 修正すべき問題の特定
  - 例: 「○○画面で△△すると落ちる」

【効果的なフィードバックを得るコツ】

1. 具体的な質問をする
   ❌「使いやすかったですか？」
   ✅「問題文は読みやすかったですか？」

2. 自由記述欄を設ける
   気づかなかった問題が見つかる

3. テスト項目を明示する
   「○○機能を試してください」
```

### テスト期間の設計

```
【推奨テスト期間】

最小: 3日間
  - 基本的な動作確認
  - 重大バグの発見

推奨: 1週間
  - 平日と週末の使用パターン
  - 継続使用での問題発見

最大: 2週間
  - 十分なフィードバック収集
  - ただし長すぎるとテスターの熱意が下がる

【テスト期間中にやること】

Day 1-2: 初期反応の確認
  - インストール成功率
  - ファーストインプレッション

Day 3-5: 詳細フィードバック収集
  - 使い込んでの感想
  - 具体的な改善点

Day 6-7: 修正とまとめ
  - 重大バグの修正
  - 最終確認ビルド
```

## 8.2 TestFlight への提出

### EAS Submit でアップロード

```bash
# TestFlight に提出
eas submit --platform ios --latest
```

### 必要な情報

| 項目 | 値 |
|------|-----|
| Apple ID | your-apple-id@example.com |
| App Store Connect App ID | 自動検出または手動入力 |

### 初回提出時の追加手順

1. App Store Connect でアプリを作成していない場合:
   ```bash
   # アプリ作成も含めて実行
   eas submit --platform ios --latest
   # プロンプトに従ってアプリ情報を入力
   ```

2. 既存アプリに提出:
   - App Store Connect の App ID を `eas.json` に設定済みであること

| フィードバックメール | support@example.com |
| デモアカウント | 不要（デバイスIDベース認証） |

以下の質問に回答:

1. **暗号化を使用していますか？**
   - はい（HTTPS通信を使用）

2. **暗号化の種類**
   - 標準の暗号化のみ（HTTPS）

3. **免除カテゴリ**
   - 一般に入手可能な暗号化ソフトウェア

## 8.4 テスター招待

### 内部テスター（最大100名）

Apple Developer チームメンバーを招待:

1. App Store Connect → ユーザーとアクセス
2. メンバーを追加
3. TestFlight → 内部テスト → テスターを追加

### 外部テスター（最大10,000名）

一般ユーザーを招待（Apple によるレビュー後に配信可能）:

1. TestFlight → 外部テスト
2. グループを作成
3. テスターを追加（メールアドレス）または公開リンクを作成

### テスター招待文テンプレート

```
【テスト参加のお願い】コンクリート診断士試験対策アプリ

開発中のアプリのベータテストにご協力ください。

【参加方法】
1. iPhone/iPad に TestFlight アプリをインストール
2. 以下のリンクからテストに参加
   [リンクURL]

【テスト期間】
〇月〇日 - 〇月〇日

【お願いしたいこと】
- アプリを使ってみての感想
```

## 8.5 フィードバック収集

### TestFlight フィードバック

テスターは TestFlight アプリ内から直接フィードバックを送信可能。

### Google Form 併用（推奨）

より詳細なフィードバックを収集:

```

5. 改善してほしい点（自由記述）


6. 良かった点（自由記述）


7. その他コメント（自由記述）

```

## 8.6 バグ対応フロー

### バグ報告時の対応

```
1. バグ報告受領
   ↓
2. 再現確認
   ↓
3. 重要度判定
   - Critical: 即座に修正
   - Major: 本リリース前に修正
   - Minor: 次バージョンで対応
   ↓
4. 修正実施
   ↓
5. 新ビルド作成・提出
   ↓
6. テスターに再テスト依頼
```

### 新ビルドの提出

```bash
# バージョン番号を更新（app.json）
# buildNumber を +1

# 再ビルド
eas build --platform ios --profile production

# 再提出
eas submit --platform ios --latest
```
