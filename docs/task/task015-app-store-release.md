## 目的

App Store Connect で申請情報を入力し、審査に提出する。審査通過後、App Store に公開する。

---

## 📚 学習ポイント：App Store 審査の全体像

### なぜ審査があるのか？

```
【Apple の審査哲学】

「ユーザーにとって安全で質の高いアプリのみを提供する」

審査の目的:
  1. セキュリティ: マルウェア、個人情報の不正取得を防ぐ
  2. 品質: クラッシュする、機能しないアプリを排除
  3. コンテンツ: 不適切・違法なコンテンツを防ぐ
  4. ルール遵守: Apple のガイドラインに従っているか

【審査があることのメリット】
✅ ユーザーが安心してアプリをダウンロードできる
✅ App Store 全体の信頼性向上
✅ 低品質アプリとの差別化ができる

【審査があることのデメリット】
❌ リリースまで時間がかかる
❌ リジェクトされる可能性がある
❌ 急な修正をすぐに反映できない
```

### 審査の流れ

```
【審査ステータスの遷移】

┌──────────┐     ┌──────────┐     ┌──────────┐
│  提出済み │────>│ 審査待ち  │────>│  審査中   │
└──────────┘     └──────────┘     └──────────┘
                                       │
              ┌────────────────────────┼────────────────┐
              ↓                        ↓                ↓
        ┌──────────┐          ┌──────────┐      ┌──────────┐
        │ リジェクト │          │   承認    │      │  保留中   │
        └──────────┘          └──────────┘      └──────────┘
              │                        │
              │                        ↓
              │               ┌──────────────────┐
              │               │ App Store 公開   │
              │               └──────────────────┘
              │
              ↓
        ┌──────────────┐
        │修正して再提出 │
        └──────────────┘

【審査期間の目安】

初回審査: 24時間〜7日（平均2-3日）
  - 新規アプリは詳しく見られる
  - 週末は処理が遅れる可能性

更新審査: 24時間〜3日
  - 既存アプリは比較的早い
  - 大きな変更は時間がかかる

緊急審査: リクエスト可能
  - 重大なバグ修正
  - セキュリティ問題
  - 期間限定イベント対応
```

### App Store Review Guidelines のポイント

```
【必ず守るべき5大ルール】

1. セーフティ（安全性）
   - クラッシュしないこと
   - ユーザーを騙さないこと
   - 個人情報を適切に扱うこと

2. パフォーマンス
   - 最小限の機能を持つこと
   - ベータ版、テスト版は NG
   - 正常に動作すること

3. ビジネス
   - App 内課金は Apple の仕組みを使う
   - 価格を明確に表示
   - 外部決済への誘導は禁止

4. デザイン
   - Apple の HIG に従う
   - 他アプリのコピーは NG
   - 機能がない「Webビュー」だけのアプリは NG

5. 法的事項
   - 著作権を侵害しない
   - プライバシーポリシーを用意
   - 年齢制限を正しく設定

【MVP アプリで特に注意すべき点】

✅ プライバシーポリシーのURL（必須）
✅ サポートURL（必須）
✅ 機能が「最小限」でも何かしら価値がある
✅ スクリーンショットと実際の画面が一致
```

## 📚 学習ポイント：リジェクトへの心構え

### リジェクトは珍しくない

```
【リジェクト統計（業界推定）】

初回申請でのリジェクト率: 約30-40%
  - ほとんどは軽微な問題
  - 修正して再申請すれば通過

【リジェクトされても大丈夫な理由】

1. 何度でも再申請できる
   - 回数制限なし
   - 無料で再審査

2. 理由が明確に伝えられる
   - Resolution Center でやり取り
   - 不明点は質問可能

3. 再審査は早い
   - 通常24-48時間
   - 同じ問題でなければスムーズ

【ネガティブに捉えない】

リジェクト ≠ アプリが悪い
リジェクト = 改善点の指摘

→ ガイドラインを学ぶ機会として活用
```

### よくあるリジェクト理由と対策

```
【Top 5 リジェクト理由】

1. Guideline 2.1 - アプリの完成度
   問題: クラッシュする、機能しない
   対策: TestFlight で十分テスト

2. Guideline 2.3 - 正確なメタデータ
   問題: スクリーンショットが実際と異なる
   対策: 最新UIで再撮影

3. Guideline 5.1 - プライバシー
   問題: プライバシーポリシーがない/不十分
   対策: 収集データを明確に記載したページを用意

4. Guideline 4.2 - 最小機能
   問題: 機能が少なすぎる
   対策: 審査メモで価値を説明

5. Guideline 1.2 - ユーザー生成コンテンツ
   問題: 不適切コンテンツ対策がない
   対策: 報告機能、ブロック機能を実装
        ※MVPは UGC なしなので関係ない

【MVP でのリジェクトリスク分析】

低リスク:
  ✅ UGC なし
  ✅ 課金なし
  ✅ 外部サービス連携なし

注意すべき:
  ⚠️ プライバシーポリシー完備
  ⚠️ 最小機能の説明
  ⚠️ スクリーンショット一致
```

## 📚 学習ポイント：リリース後の運用

### App Store Connect Analytics

```
【見るべき指標】

1. インプレッション数
   - App Store で表示された回数
   - ASO の効果測定

2. プロダクトページ閲覧数
   - 詳細ページを見た回数
   - アイコンやスクリーンショットの効果

3. ダウンロード数
   - App Unit（初回ダウンロード）
   - 再ダウンロード

4. コンバージョン率
   - 閲覧数 → ダウンロード数
   - 業界平均: 約30%

【改善サイクル】

インプレッション低い → ASO改善（キーワード）
閲覧数低い → アイコン改善
コンバージョン低い → スクリーンショット・説明文改善
```

### レビュー対応

```
【レビューの重要性】

App Store での評価:
  ★★★★★ (4.5以上) → 検索順位アップ、信頼性高い
  ★★★★☆ (4.0-4.4) → 良好
  ★★★☆☆ (3.0-3.9) → 改善が必要
  ★★☆☆☆ (2.0-2.9) → 要注意
  ★☆☆☆☆ (1.0-1.9) → 深刻な問題あり

【レビューへの返信のコツ】

✅ すべてのレビューに返信（特に低評価）
✅ 感謝を伝える
✅ 問題には具体的な対応を説明
✅ 改善予定があれば伝える

例:
「ご利用いただきありがとうございます。
 文字サイズの件、次回アップデートで
 調整機能を追加予定です。
 引き続きよろしくお願いいたします。」

【低評価レビューの対処】

1. 感情的にならない
2. 問題を認識していることを伝える
3. 対応策を具体的に説明
4. 改善したら再評価をお願い
```

### 継続的な改善サイクル

```
【アプリ運用のPDCAサイクル】

Plan（計画）
  ↓ ユーザーフィードバック、Analytics から課題を特定
Do（実行）
  ↓ 新機能開発、バグ修正
Check（確認）
  ↓ TestFlight でテスト
Act（改善）
  ↓ App Store 更新リリース
  ↓
（繰り返し）

【リリース頻度の目安】

緊急修正: 即時
  - クラッシュバグ
  - セキュリティ問題

定期アップデート: 2-4週間ごと
  - 機能改善
  - バグ修正
  - iOS 対応

メジャーアップデート: 3-6ヶ月ごと
  - 新機能追加
  - UI刷新
```

| 項目 | 入力内容 |
|------|---------|
| バージョン | 1.0.0 |

| 項目 | 入力内容 |

## 9.2 App Privacy 設定

### データ収集の宣言

App Store Connect → 「App のプライバシー」

#### 収集するデータ

| データタイプ | 収集する | 目的 | ユーザーにリンク |
|-------------|---------|------|-----------------|
| 識別子 → デバイスID | はい | アプリの機能 | いいえ |
| 使用状況データ → 製品の操作 | はい | アプリの機能 | いいえ |
| 診断 → クラッシュデータ | はい | アプリの機能 | いいえ |

#### トラッキング

「いいえ」を選択（他社のアプリやWebサイトでのトラッキングは行わない）

## 9.4 審査に提出

### 審査情報

| 項目 | 入力内容 |
|------|---------|
| サインインが必要 | いいえ |
| 連絡先情報 | 名前、電話番号、メール |
| メモ（審査担当者向け） | 任意 |

### 審査メモの例

```
本アプリは、コンクリート診断士試験の学習支援アプリです。

【主な機能】
- 過去問100問の問題演習
- 正誤判定と解説表示
- 学習進捗の記録

【テスト方法】
1. アプリを起動
2. 年度を選択
3. 問題を選択して解答
4. 結果と解説を確認

サインインは不要です。デバイスIDで自動的にユーザー管理されます。
```

### 提出

「審査に提出」ボタンをクリック

## 9.5 審査プロセス

### 審査ステータス

| ステータス | 説明 | 期間 |
|-----------|------|------|
| 提出済み | 審査キューに入った | - |
| 審査待ち | 審査開始を待っている | 1-3日 |
| 審査中 | 審査担当者が確認中 | 1-2日 |
| 承認 | 審査通過 | - |
| リジェクト | 問題あり・要修正 | - |

### 審査期間の目安

- 初回審査: 1-7日（平均2-3日）
- 再審査: 1-3日

## 9.6 リジェクト対応

### よくあるリジェクト理由と対策

| 理由 | 対策 |
|------|------|
| **プライバシーポリシー不備** | URLが有効か確認、内容を充実 |
| **スクリーンショット不一致** | 最新UIで再撮影 |
| **クラッシュ** | 該当端末でテスト、修正後再提出 |
| **最小機能要件** | 機能説明を審査メモに追記 |
| **不適切なコンテンツ** | 該当箇所を削除・修正 |
| **メタデータの問題** | 説明文・キーワードを修正 |

### リジェクト時のフロー

```
1. リジェクト理由を確認
   ↓
2. 問題箇所を特定
   ↓
3. 修正実施
   ↓
4. 新ビルド作成（必要な場合）
   ↓
5. 再提出
   ↓
6. Resolution Center でメッセージ（必要に応じて）
```

### Resolution Center

審査担当者とコミュニケーションが必要な場合:
1. App Store Connect → アプリ → 審査
2. 「Resolution Center」
3. メッセージを送信

## 9.7 審査通過後

### リリース方法の選択

| オプション | 説明 |
|-----------|------|
| 手動リリース | 承認後、手動で公開 |
| 自動リリース | 承認後、即座に公開 |
| 日時指定 | 指定日時に自動公開 |

### リリース実行

「手動リリース」の場合:
1. App Store Connect → バージョン
2. 「リリース」ボタンをクリック

### リリース後の確認

- [ ] App Store で検索して表示される
- [ ] 自分のデバイスでダウンロード可能
- [ ] アプリが正常に起動・動作

---

## 9.8 リリース後タスク

### 必須対応

- [ ] ダウンロード・動作確認
- [ ] SNSで公開告知（オプション）
- [ ] サポート体制の準備

### モニタリング

- [ ] App Store Connect Analytics 確認
- [ ] クラッシュレポート監視
- [ ] ユーザーレビュー確認

### 緊急対応準備

問題発生時の緊急リリース手順:
1. 修正実施
2. バージョン番号更新（1.0.1）
3. eas build → eas submit
4. 「緊急審査リクエスト」（該当する場合）
