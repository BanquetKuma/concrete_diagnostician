## 📚 学習ポイント：ソフトウェアテストの基礎

### なぜテストが重要か？

```
【テストなしでリリースすると...】

開発完了 → そのままリリース → ユーザーがバグ発見 → 悪いレビュー
    ↓
App Store 評価 ★1.5 💀
    ↓
ダウンロード激減 → アプリ終了

【テストしてからリリースすると...】

開発完了 → テスト → バグ発見・修正 → リリース → 良い体験
    ↓
App Store 評価 ★4.5 ⭐
    ↓
口コミ拡散 → 成功！
```

### テストの種類（テストピラミッド）

```
          △
         /  \
        / E2E\      ← エンドツーエンドテスト（実際のユーザー操作を模倣）
       /──────\
      /  統合   \    ← 統合テスト（複数の部品が連携して動くか）
     /────────────\
    /    単体テスト  \  ← ユニットテスト（個々の関数・コンポーネント）
   /──────────────────\

【ピラミッドの意味】
下に行くほど:
- テスト数が多い
- 実行速度が速い
- コストが低い

上に行くほど:
- テスト数が少ない
- 実行速度が遅い
- コストが高い
```

このタスクでは主に**E2Eテスト（手動）**と**統合テスト**に焦点を当てます。

### MVP での手動テストの位置づけ

```
【自動テスト vs 手動テスト】

自動テスト:
  ✅ 何度でも同じテストを実行可能
  ✅ リグレッション（退行）を防げる
  ❌ 初期実装コストが高い
  ❌ UI変更のたびに修正が必要

手動テスト:
  ✅ 柔軟にテスト可能
  ✅ 実装コストが低い
  ❌ 人的ミスが起きやすい
  ❌ 繰り返しが大変

【MVPでの戦略】
最初は手動テスト中心 → ユーザー獲得後に自動テスト導入
（最初から完璧を目指さず、素早くリリースする）
```

## 📚 学習ポイント：機能テストの考え方

### テストケースの設計

```
【良いテストケース】
1. 正常系（Happy Path）: 期待通りの入力で期待通りの結果
2. 異常系（Unhappy Path）: 想定外の入力でも壊れない
3. 境界値（Boundary）: 最小値・最大値付近

例：「選択肢をタップする」のテスト

正常系: ボタンを1回タップ → 選択が確定される
異常系: 素早く2回タップ → 2回目は無視される
境界値: (このケースでは該当なし)
```

### 「期待動作」と「結果」の書き方

```
【曖昧な期待動作】❌
「正しく動作する」
「エラーが出ない」
「ちゃんと表示される」

【具体的な期待動作】✅
「ボタンタップ後、緑色 (#4CAF50) で正解がハイライトされる」
「APIエラー時、"接続できません"というメッセージが表示される」
「50問中20問解答済みの場合、"20 / 50" と表示される」
```

## 📚 学習ポイント：モバイルUI設計の基本

### タップ領域（Hit Target）の重要性

```
【Appleのガイドライン】
最小タップ領域: 44pt × 44pt

なぜ 44pt か？
- 人間の指の平均的なタッチ面積に基づく
- これより小さいと押し間違いが増える

【悪い例】         【良い例】
  [小さいボタン]     [    大きいボタン    ]
       ↑                    ↑
   押しにくい！          押しやすい！

実装のコツ:
- ボタンのpaddingを十分に取る
- 視覚的には小さくても、タップ領域は44pt確保
```

### Safe Area（セーフエリア）とは？

```
【iPhone の画面構造】

┌─────────────────────────┐
│ ステータスバー (時計・電波)  │ ← Safe Areaの外
├─────────────────────────┤
│                         │
│                         │
│      Safe Area          │ ← ここにコンテンツを配置
│    （安全な領域）         │
│                         │
│                         │
├─────────────────────────┤
│       ホームバー         │ ← Safe Areaの外
└─────────────────────────┘

【ノッチ付きデバイス】

┌─────────┬───┬─────────┐
│         │███│         │ ← ノッチ (カメラ・センサー)
│                         │
│      Safe Area          │
│                         │
│                         │
└─────────────────────────┘

React Native での対応:
import { SafeAreaView } from 'react-native-safe-area-context';

<SafeAreaView>
  {/* ここにコンテンツ */}
</SafeAreaView>
```

### コントラストとアクセシビリティ

```
【WCAGガイドライン】
テキストと背景のコントラスト比:
- 通常テキスト: 4.5:1 以上
- 大きいテキスト: 3:1 以上

【具体例】
✅ 良い: 白背景 (#FFFFFF) に 黒文字 (#000000) → 21:1
✅ 良い: 白背景 (#FFFFFF) に グレー (#757575) → 4.6:1
❌ 悪い: 白背景 (#FFFFFF) に 薄グレー (#CCCCCC) → 1.6:1

チェックツール:
https://webaim.org/resources/contrastchecker/
```

## 5.3 パフォーマンステスト

> 💡 **パフォーマンスが重要な理由**
>
> - **1秒の遅延** → コンバージョン率が **7%低下**（Amazon調査）
> - **3秒以上の読み込み** → **53%のユーザーが離脱**（Google調査）
>
> ユーザーは「遅い」と感じたら、すぐに別のアプリに乗り換えます。

### 目標値

| 指標 | 目標値 | 測定方法 |
|------|--------|---------|
| 初回起動 | < 3秒 | ストップウォッチ |
| 画面遷移 | < 300ms | 体感 |

## 📚 学習ポイント：パフォーマンス測定の方法

### React Native のパフォーマンス測定

```
【開発者ツールの使い方】

1. Expo DevTools (ブラウザ)
   - npx expo start → ブラウザで開く
   - Performance タブで確認

2. React DevTools
   - npm install -g react-devtools
   - react-devtools コマンドで起動
   - Profiler タブでレンダリング時間を確認

3. Flipper (より詳細な分析)
   - React Native の公式デバッグツール
   - ネットワーク、レイアウト、データベースの確認
```

### メモリリークとは？

```
【メモリリーク】
使い終わったメモリが解放されず、どんどん消費量が増える現象

【よくある原因】
1. useEffect の cleanup忘れ
   ❌ 悪い例:
   useEffect(() => {
     const timer = setInterval(() => {...}, 1000);
     // cleanup なし → タイマーが動き続ける
   }, []);

   ✅ 良い例:
   useEffect(() => {
     const timer = setInterval(() => {...}, 1000);
     return () => clearInterval(timer); // cleanup
   }, []);

2. イベントリスナーの解除忘れ
3. 大きな配列/オブジェクトの保持

【チェック方法】
長時間使用後のメモリ ÷ 起動直後のメモリ
→ 2倍以上なら要調査
```

## 📚 学習ポイント：オフライン対応の考え方

### グレースフルデグラデーション

```
【Graceful Degradation（優雅な縮退）】
ネットワークが切れても、可能な範囲で機能を維持する考え方

レベル1: 完全動作
  - すべての機能が正常に動作

レベル2: 縮退動作
  - キャッシュデータで表示
  - 新規データの取得は不可

レベル3: 最小動作
  - エラーメッセージを表示
  - リトライボタンを提供

【MVPでの対応】
- 現在表示中のデータは保持
- 新規取得は「接続を確認してください」と表示
- 過度な対応は後回し（オフラインファースト設計は複雑）
```

## 5.6 バグ修正トラッキング

## 📚 学習ポイント：バグの分類と優先度

### バグの重要度（Severity）

```
【重要度の定義】

致命的 (Critical):
  - アプリがクラッシュする
  - データが消失する
  - セキュリティに問題がある
  例: 解答を保存するとアプリが落ちる

重大 (Major):
  - 主要機能が使えない
  - 回避策がない
  例: 問題が表示されない

中程度 (Moderate):
  - 機能に問題があるが回避策あり
  例: 「次の問題」ボタンが効かないが、一覧に戻れば次に進める

軽微 (Minor):
  - 見た目の問題
  - 使用に支障なし
  例: フォントサイズが若干小さい
```

### バグの優先度（Priority）

```
【優先度の決め方】

優先度 = 重要度 × 発生頻度

        ┌────────────────────────────────────┐
        │         発生頻度                   │
        │   高い      中程度      低い       │
        ├────────────────────────────────────┤
重  致命的│  P1 即修正  P1 即修正  P2 早急に  │
要  重大  │  P1 即修正  P2 早急に  P3 計画的  │
度  中程度│  P2 早急に  P3 計画的  P4 余裕時  │
    軽微  │  P3 計画的  P4 余裕時  P5 保留可  │
        └────────────────────────────────────┘

【リリース判断】
- P1, P2 のバグ → リリース前に必ず修正
- P3 のバグ → 可能であれば修正
- P4, P5 のバグ → 次回リリースで対応可
```
