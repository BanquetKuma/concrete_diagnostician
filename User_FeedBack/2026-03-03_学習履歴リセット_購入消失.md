## 調査経緯

### 初期仮説: 異なるアカウントでのログイン (Scenario A) → ❌ 否定

ユーザーから追加情報:
- Googleログインはブロックされる → Appleでログインしている
- 1つのアカウントでしかログインしていない

→ 異なるプロバイダ切り替えは原因ではない

### 確定した原因: 購入が匿名IDに紐付いた (RevenueCat logIn 問題)

**RevenueCat ダッシュボード調査結果**:

| 項目 | 値 |
|------|-----|
| Clerk User ID | `user_38se3IZwrtevYyvXwVUOMwr7heV` |
| RevenueCat での Clerk User ID | 存在するが entitlement なし |
| 購入の紐付け先 | `$RCAnonymousID` (匿名ID) |
| 購入製品 | `com.banquetkuma.concretediagnostician.full` (買い切り JPY 300) |
| 月額サブスク | 2026-01-25 購入 → 2026-02-13 解約 → 2026-02-25 期限切れ |

**根本原因**:
`Purchases.logIn(clerkUserId)` が購入時に実行されていない or 失敗していた。
そのため購入は `$RCAnonymousID` に紐付き、Clerk User ID でログインすると別顧客扱いとなり、
購入が見えなくなった。`restorePurchases()` も Clerk User ID コンテキストでは購入が見つからない。

### 副次的要因: userService.ts のフォールバックバグ

修正前の `userService.ts` の `catch` ブロックで新しい device ID を生成して既存 ID を上書きするバグがあり、
学習履歴の消失にも寄与した可能性がある（v1.0.3 で修正済み）。

```bash
  -H "Content-Type: application/json" \
  -d '{"duration": "lifetime"}'
```

### Phase 3: 根本修正（v1.0.4 予定）

`contexts/RevenueCatContext.tsx` に `ensureLoggedIn()` ガードを追加。
購入・復元の前に `Purchases.logIn(clerkUserId)` を必ず呼ぶことで、
匿名IDへの購入紐付けを防止。

```typescript
const ensureLoggedIn = useCallback(async () => {
  if (clerkUserId) {
    try {
      await Purchases.logIn(clerkUserId);
    } catch (err) {
      console.error('RevenueCat logIn failed:', err);
    }
  }
}, [clerkUserId]);
```

## RevenueCat トラブルシューティング

### 購入が匿名IDに紐付いた場合の対応手順

1. **RevenueCat ダッシュボード** → Customers → Clerk User ID で検索
2. ヒットした場合 → active_entitlements を確認
3. entitlement なし → 購入が匿名IDに紐付いている
4. **V1 API で手動 Grant**:

```bash
# V1 Secret API key が必要（V2 key では動かない！）
curl -X POST \
  "https://api.revenuecat.com/v1/subscribers/{CLERK_USER_ID}/entitlements/{ENTITLEMENT_LOOKUP_KEY}/promotional" \
  -H "Authorization: Bearer {V1_SECRET_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{"duration": "lifetime"}'
```

### 注意点
- RevenueCat **V2 API key** では V1 の promotional grant エンドポイントは使えない → V1 key を別途生成
- RevenueCat ダッシュボードには「顧客の新規作成」ボタンがない → API 経由のみ
- RevenueCat の Transaction Details には **Apple Order ID は表示されない**（transaction_id のみ）
- Order ID での検索は RevenueCat では不可 → Clerk User ID or 匿名ID で検索

### RevenueCat API リファレンス

| 操作 | API Version | エンドポイント |
|------|------------|---------------|
| エンタイトルメント一覧 | V2 | `GET /v2/projects/{PROJECT_ID}/entitlements` |
| 顧客情報取得 | V2 | `GET /v2/projects/{PROJECT_ID}/customers/{APP_USER_ID}` |
| Promotional Grant | **V1 のみ** | `POST /v1/subscribers/{APP_USER_ID}/entitlements/{LOOKUP_KEY}/promotional` |

duration options: `daily`, `three_day`, `weekly`, `monthly`, `two_month`, `three_month`, `six_month`, `yearly`, `lifetime`

## 教訓

1. **RevenueCat + Clerk 連携では、購入前の logIn 完了を保証すること**
2. **RevenueCat の検索は Order ID 非対応** — Clerk User ID or App User ID で検索
3. **V1/V2 API の機能差異に注意** — Promotional Grant は V1 のみ
4. **匿名IDでの購入は `restorePurchases()` では復元不可** — 手動 Grant が必要
